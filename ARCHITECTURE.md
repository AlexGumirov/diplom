# Архитектура проекта

Проект разделен на два основных слоя:

- `monitoring/` — Django backend, API, модели, бизнес-логика и аналитика.
- `ui/` — React frontend, который собирается в `monitoring/static/ui/` и отдается Django как SPA.

## Backend

```text
monitoring/
  models.py              # ORM-модели и связи таблиц
  serializers.py         # DRF-сериализаторы и валидация API
  views.py               # REST API ViewSet'ы
  frontend_views.py      # Тонкий HTTP-слой для SPA и app-api
  frontend_urls.py       # Маршруты пользовательского интерфейса
  presenters.py          # Форматирование данных backend -> frontend
  services/
    analysis.py          # Аналитика, дельта-анализ, корреляции, аномалии
    calculations.py      # Расчет интегральных оценок
    normalization.py     # Нормализация показателей к шкале 1-10
    record_workflow.py   # Сценарии сохранения физики и САН
    san.py               # Вопросы и расчет методики САН
  utils/
    validators.py        # Общие валидаторы и парсинг значений
```

## Frontend

```text
ui/src/
  app/
    api.ts               # Запросы к Django app-api
    types.ts             # Общие типы frontend/backend контракта
    App.tsx              # Навигация и загрузка bootstrap-данных
    components/          # Страницы и UI-компоненты
    lib/records.ts       # Преобразование DailyRecord в записи дневника
  styles/                # Глобальные стили и тема
```

## Рабочий пользовательский сценарий

1. Пользователь открывает SPA через Django.
2. Frontend загружает `/app-api/bootstrap/`.
3. Пользователь сохраняет физические данные через `/app-api/physical/`.
4. Пользователь проходит САН и сохраняет ответы через `/app-api/san/`.
5. Backend пересчитывает оценки, если за дату есть и физические, и психологические данные.
6. Frontend показывает записи, графики и дельта-анализ по сохраненным `StateScore`.

## Реализованный функционал

### Пользователь и доступ

- Авторизация через стандартную Django-сессию.
- Страница входа `/login/`.
- Выход через `/logout/`.
- Все страницы SPA и `app-api` доступны только авторизованному пользователю.
- Данные API фильтруются по текущему пользователю.

### Профиль спортсмена

- Для каждого пользователя автоматически создается `AthleteProfile`.
- Профиль хранит возраст, пол и вид спорта.
- Пол профиля поддерживает только два значения: мужской и женский.
- Профиль отображается в SPA.
- Редактирование профиля подключено к backend через `/app-api/profile/`.
- Возраст профиля валидируется в диапазоне 5-100 лет.

### Дневные записи

- `DailyRecord` создается на конкретную дату.
- Для одного профиля может быть только одна запись на дату.
- Запись объединяет физические данные, психологические данные и итоговые оценки.

### Физические показатели

Пользователь может сохранить:

- сон в часах;
- количество приемов пищи;
- ЧСС в покое;
- ЧСС при нагрузке;
- время восстановления;
- усталость;
- RPE.

Backend валидирует допустимые диапазоны и умеет принимать время восстановления как число минут или строку `мм:сс`.

Заполнение данных в SPA сохраняется как единая дневная запись: физические показатели, ответы САН и заметка отправляются одним запросом на `/app-api/records/`.
Заметки хранятся в `DailyRecord.notes`, возвращаются в bootstrap-ответе и отображаются в дневнике записей.
Для одной даты у спортсмена допускается только одна запись; повторное заполнение той же даты и даты из будущего отклоняются backend-валидацией.

### Психологические показатели САН

- В SPA реализовано прохождение 30 вопросов САН.
- Ответы сохраняются в `SANTest` и `SANAnswer`.
- На основе ответов создается или обновляется `PsychologicalData`.
- Для вопросов, где положительное состояние находится справа, оценка инвертируется при расчете.

### Итоговые оценки

- Если за дату есть и физические, и психологические данные, backend рассчитывает:
  - `physical_score`;
  - `psychological_score`;
  - `total_score`.
- Итог сохраняется в `StateScore`.
- В основном SPA-сценарии физика и САН сохраняются вместе, поэтому итоговая оценка появляется сразу после успешного сохранения полной записи.
- Старые endpoints раздельного сохранения физики и САН оставлены для совместимости.

### Отображение в SPA

- Профиль.
- Психология.
- Физика.
- Общее состояние.
- Заполнение данных.
- Дневник записей.
- Просмотр деталей записи.
- Дельта-анализ между двумя последними записями с итоговой оценкой.
- Агрегированный дельта-анализ за период.
- Дельта-анализ между соседними днями.
- Для страниц физики, психологии и общего состояния дельта-анализ считается по соответствующей оценке: `physical_score`, `psychological_score`, `total_score`.
- Страница статистики использует реальный корреляционный анализ Пирсона по данным пользователя.
- Страница статистики показывает критические отклонения последней записи через Isolation Forest и сравнение со средними значениями спортсмена.
- Страница заполнения данных не сохраняет физику отдельно: пользователь заполняет физические поля, проходит САН и нажимает одну финальную кнопку `Сохранить данные`.
- После сохранения или ошибки отображается всплывающее уведомление; клиентская и backend-валидация проверяют диапазоны, заполненность САН, запрет будущей даты и запрет повторной даты.

### DRF API

Дополнительно к SPA API есть REST API `/api/`:

- профили спортсменов;
- дневные записи;
- физические данные;
- психологические данные;
- итоговые оценки;
- аналитика с корреляциями, аномалиями и дельтой.

### SPA API полной записи

Основной endpoint заполнения данных:

```text
POST /app-api/records/
```

Payload:

```json
{
  "date": "2026-05-14",
  "notes": "Комментарий к тренировке",
  "physical_data": {
    "sleep_hours": 8,
    "meals": 3,
    "heart_rate_rest": 60,
    "heart_rate_load": 140,
    "recovery_time": "2:10",
    "fatigue": 5,
    "rpe": 5
  },
  "answers": [
    { "question_number": 1, "value": 4 }
  ]
}
```

Сохранение выполняется атомарно:

- создается `DailyRecord` с `notes`;
- валидируются и сохраняются `PhysicalData`;
- сохраняются ответы САН и рассчитываются `PsychologicalData`;
- рассчитывается `StateScore`;
- при ошибке ни одна часть записи не остается частично сохраненной.

Ошибки даты:

```text
date > today -> 400
existing DailyRecord(profile, date) -> 400
```

## Архитектурные замечания

- В проекте есть два API-слоя: DRF `/api/` и отдельный SPA API `/app-api/`. Это допустимо для MVP, но часть логики может дублироваться. В дальнейшем лучше явно закрепить роли: DRF как публичный ресурсный API, `app-api` как агрегирующий слой для экранов.
- Поле заметок в форме физики есть в UI, но в модели и API такого поля нет.
- `SECRET_KEY`, `DEBUG=True` и `ALLOWED_HOSTS=["*"]` подходят только для локальной разработки.
- `db.sqlite3` и `__pycache__` не должны храниться в git-истории как рабочие артефакты.
- Добавлены unit-тесты для нормализации, САН, интегральных оценок, дельта-анализа, корреляций и редактирования профиля. Права доступа к API еще стоит покрыть шире отдельными тестами.
- Bundle frontend собирается успешно, но `app.js` больше 500 KB. Позже можно добавить code splitting.

## Формулы расчетов

Все итоговые нормализованные физические и психологические оценки приводятся к шкале от 1 до 10.

### Ограничение диапазона

```text
clamp(x, min, max) = max(min, min(max, x))
```

### Прямая нормализация

Используется, когда большее исходное значение считается лучшим:

```text
X_norm = 1 + 9 * (X - X_min) / (X_max - X_min)
```

После расчета значение ограничивается диапазоном 1-10.

### Обратная нормализация

Используется, когда меньшее исходное значение считается лучшим:

```text
X_norm = 1 + 9 * (X_max - X) / (X_max - X_min)
```

После расчета значение ограничивается диапазоном 1-10.

### Физические показатели

```text
sleep_norm = 1 + 9 * (sleep_hours - 0) / (12 - 0)
meals_norm = 1 + 9 * (meals - 0) / (5 - 0)
heart_rate_rest_norm = 1 + 9 * (90 - heart_rate_rest) / (90 - 50)
heart_rate_load_norm = 1 + 9 * (220 - heart_rate_load) / (220 - 90)
recovery_norm = 1 + 9 * (5 - recovery_time) / (5 - 1)
fatigue_norm = 1 + 9 * (10 - fatigue) / (10 - 1)
rpe_norm = 1 + 9 * (10 - rpe) / (10 - 1)
```

Каждая нормализованная оценка ограничивается диапазоном 1-10.

Физическая оценка:

```text
physical_score = average([
  sleep_norm,
  meals_norm,
  heart_rate_rest_norm,
  heart_rate_load_norm,
  recovery_norm,
  fatigue_norm,
  rpe_norm
])
```

Результат округляется до двух знаков.

### САН

Ответы САН хранятся в шкале 1-7.

Для прямых вопросов значение используется как есть:

```text
san_value = answer_value
```

Для обратных вопросов значение инвертируется:

```text
san_value = 8 - answer_value
```

Обратные вопросы:

```text
3, 4, 9, 10, 13, 15, 16, 21, 22, 27, 28
```

Группы САН:

```text
wellbeing = average(1, 2, 7, 8, 13, 14, 19, 20, 25, 26)
activity = average(3, 4, 9, 10, 15, 16, 21, 22, 27, 28)
mood = average(5, 6, 11, 12, 17, 18, 23, 24, 29, 30)
```

Интерпретация САН:

```text
score < 4      -> плохое состояние
4 <= score <= 5.5 -> нормальное состояние
score > 5.5    -> хорошее состояние
```

### Психологическая оценка

Каждый показатель САН переводится из шкалы 1-7 в шкалу 1-10:

```text
san_norm = 1 + 9 * (san_score - 1) / (7 - 1)
```

Психологическая оценка:

```text
psychological_score = average(wellbeing_norm, activity_norm, mood_norm)
```

Результат округляется до двух знаков.

### Общая оценка

```text
total_score = (physical_score + psychological_score) / 2
```

Результат округляется до двух знаков.

### Дельта-анализ

Сравниваются две последние записи, у которых есть `StateScore`:

```text
delta = current_total_score - previous_total_score
```

Интерпретация:

```text
delta > 0   -> improvement
delta < 0   -> deterioration
delta = 0   -> stable
```

Если двух записей с итоговой оценкой нет, возвращается `insufficient_data`.

### Агрегированный Дельта-Анализ

Для периода из `n` записей с итоговой оценкой:

```text
delta_G_n = sum(G_i - G_(i-1)), i = 2..n
```

В интерфейсе поддерживаются периоды:

```text
7 дней, 14 дней, 31 день, весь период, соседние дни
```

Для обычных периодов backend возвращает:

```text
period_start
period_end
days_count
first_score
last_score
total_delta
status
```

Для режима `neighbors` дополнительно возвращается список соседних переходов:

```text
from_date
to_date
previous_score
current_score
delta
status
```

Дельта-анализ поддерживает три метрики:

```text
overall -> total_score
physical -> physical_score
psychological -> psychological_score
```

### Аналитика

Backend содержит два дополнительных аналитических метода:

- корреляционный анализ Пирсона через pandas `corr`;
- поиск аномалий через `IsolationForest`.

Корреляционный анализ доступен в SPA через `/app-api/correlations/` и в DRF через `/api/analysis/correlations/`.
Для расчета используются только полные записи, где есть физические данные, психологические данные и `StateScore`.
Минимум для анализа — 7 полных записей. Связи с `|r| < 0.3`, диагональ матрицы, дубликаты пар и `NaN` не отображаются.
В пользовательском отчете показываются только связи между исходными физическими показателями и психологическими шкалами САН:

```text
sleep_hours, meals, heart_rate_rest, heart_rate_load, recovery_time, fatigue, rpe
  сравниваются с
wellbeing, activity, mood
```

В карточках для каждой шкалы САН выбирается один физический показатель с максимальным `|r|`.
Агрегированные оценки `physical_score`, `psychological_score` и `total_score` в карточках корреляций не используются.

Интерпретация коэффициента Пирсона:

```text
abs(r) < 0.3      -> слабая связь
0.3 <= abs(r) < 0.7 -> умеренная связь
abs(r) >= 0.7     -> сильная связь
r > 0             -> прямая связь
r < 0             -> обратная связь
r = 0             -> связи нет
```

Анализ критических отклонений доступен в SPA через `/app-api/anomalies/` и в DRF через `/api/analysis/anomalies/`.
Для расчета используются только полные записи. Минимум для анализа — 7 полных записей.
Isolation Forest используется как общий индикатор нетипичности последней записи и обучается только на исходных нормализованных признаках:

```text
IsolationForest(contamination="auto", random_state=42)
```

Используемые признаки:

```text
sleep_norm
meals_norm
heart_rate_rest_norm
heart_rate_load_norm
recovery_norm
fatigue_norm
rpe_norm
wellbeing_norm
activity_norm
mood_norm
```

Агрегированные оценки `physical_score`, `psychological_score` и `total_score` в Isolation Forest и анализе критических отклонений не используются.

Слой ML и слой пользовательского отчета разделены:

- ML-слой продолжает обучать `IsolationForest` только на нормализованных признаках `*_norm`.
- UI-слой получает и показывает исходные raw-значения в привычных единицах спортсмена.
- Нормализованные значения остаются в JSON только как диагностические поля `normalized_current_value` и `normalized_mean_value`; frontend их не отображает.

Конкретные критические показатели определяются сравнением raw-значения последней записи со средним raw-значением спортсмена:

```text
raw_mean_value = average(X_1, X_2, ..., X_n)
std_value = std(X_1, X_2, ..., X_n)
raw_difference = raw_current_value - raw_mean_value
critical_threshold = max(2 * std_value, min_threshold)
abs(raw_difference) >= critical_threshold -> критическое отклонение
```

Минимальные абсолютные пороги:

```text
sleep_hours -> 1.0 ч
meals -> 1 прием пищи
heart_rate_rest -> 5 уд/мин
heart_rate_load -> 10 уд/мин
recovery_time -> 0.5 мин
fatigue -> 2 балла
rpe -> 2 балла
wellbeing -> 1 балл
activity -> 1 балл
mood -> 1 балл
```

Элемент отчета по отклонению возвращается в формате:

```json
{
  "key": "sleep_hours",
  "label": "Сон",
  "raw_current_value": 5,
  "raw_mean_value": 7.8,
  "raw_difference": -2.8,
  "normalized_current_value": 4.75,
  "normalized_mean_value": 6.58,
  "unit": "ч",
  "direction": "below",
  "direction_label": "ниже",
  "severity": "high",
  "message": "Продолжительность сна ниже среднего значения спортсмена на 2.8 часа."
}
```

Единицы измерения для UI:

```text
sleep_hours -> "ч"
meals -> ""
heart_rate_rest -> "уд/мин"
heart_rate_load -> "уд/мин"
recovery_time -> "мин"
fatigue -> "балла"
rpe -> "балла"
wellbeing -> "балла"
activity -> "балла"
mood -> "балла"
```

UX блока критических отклонений:

- На странице статистики не отображаются фильтры периода `7 дней`, `14 дней`, `31 день`; корреляционный блок загружается за весь доступный период.
- При наличии значимых raw-отклонений показывается текст: `Обнаружены показатели, значительно отличающиеся от привычного состояния спортсмена.`
- Если значимых raw-отклонений нет: `Показатели находятся в пределах привычного диапазона спортсмена.`
- Интерпретация Isolation Forest выводится отдельно:
  - `Последняя запись существенно отличается от обычного состояния спортсмена.`
  - `Последняя запись соответствует типичному состоянию спортсмена.`
- В карточках отображаются только `raw_current_value`, `raw_mean_value` и `raw_difference`.
- `recovery_time` в UI форматируется как `мм:сс`; остальные показатели выводятся с единицами из карты выше.
- Сообщения не используют агрессивное `ВНИМАНИЕ!` и не показывают нормализованные величины пользователю.

Проверки по САН и recovery:

- `activity` хранится как raw-среднее САН в шкале 1-7 и нормализуется один раз в `activity_norm` для ML.
- Повторной инверсии САН после сохранения нет: инверсия применяется при расчете результатов САН из ответов.
- Парсер recovery использует `minutes + seconds / 60`, поэтому `2:10 -> 2.1667`, `1:55 -> 1.9167`.

Корреляционные тексты в UI формулируются через `наблюдается связь` и `статистическая взаимосвязь`, без причинных утверждений вроде `влияет`.

Добавлены и обновлены тесты для:

- нового порога `max(2 * std_value, min_threshold)`;
- случая `sleep_hours` с отклонением около `-2.8 ч`;
- игнорирования малой разницы `fatigue = 0.7`;
- raw-полей в ответе `/app-api/anomalies/`;
- человекочитаемых сообщений с единицами;
- отсутствия normalized-значений в UI блока;
- отсутствия фильтров периода на странице статистики;
- сохранения заметок в дневную запись через `/app-api/records/`;
- запрета повторного заполнения одной даты;
- запрета заполнения будущей даты;
- единой финальной кнопки сохранения на странице заполнения данных;
- однократной нормализации `activity`;
- парсинга recovery `2:10` и `1:55`.
